# 密码管理工具技术方案（Node.js + Vue3 + MySQL + pnpm Monorepo）

## 1. 项目目标与范围

### 1.1 目标
构建一个以安全为首要原则的密码管理工具，支持用户对不同平台账号密码进行集中管理，并支持附加信息与标签分类，前端同时适配 PC 与 H5。

### 1.2 核心功能范围（MVP）
- 账号记录管理：新增、编辑、删除、查询、详情查看
- 记录字段：平台名、账号、密码、附加信息（备注/自定义字段）
- 标签系统：每条记录可绑定多个标签
- 搜索与过滤：按平台、账号、标签、更新时间过滤
- 安全能力：登录认证、会话管理、审计日志、数据加密

---

## 2. 总体架构

### 2.1 架构原则
- 安全优先：默认最小权限、默认加密、默认可审计
- 分层解耦：前端、API、数据层职责清晰
- 可演进：MVP 可快速上线，后续可平滑扩展多用户/共享

### 2.2 架构概览
- 前端：`Vue3 + TypeScript + Vite`，单应用响应式支持 PC/H5
- 服务端：`Node.js + TypeScript + Fastify`
- 数据库：`MySQL 5.7`
- 会话与限流：优先基于数据库与应用内中间件实现，暂不引入 Redis
- 部署形态：单机 Docker Compose（`web + api + mysql`）满足个人使用

### 2.3 核心安全设计（关键）
优先推荐“端到端加密（E2EE）”策略：
- 密码明文只在前端内存中短暂存在
- 记录敏感字段在前端加密后再上传
- 服务端与数据库仅存储密文与必要索引
- 服务端被攻破时，攻击者无法直接解密用户密码明文

---

## 3. 安全威胁模型与防护

### 3.1 主要威胁
- 数据库泄露（SQL 注入、备份泄露、运维误操作）
- 服务端入侵（RCE、依赖漏洞）
- 前端攻击（XSS、CSRF、恶意脚本注入）
- 凭证攻击（撞库、暴力破解、会话劫持）
- 内部风险（日志泄露敏感信息、开发环境泄漏）

### 3.2 安全控制清单
- 认证：`Argon2id` 哈希登录密码（高成本参数）
- 二次验证（建议）：TOTP（如 Google Authenticator）
- 会话：短期 Access Token + 可轮换 Refresh Token
- 传输：全站 HTTPS + HSTS + TLS1.2+
- 输入校验：服务端全量 schema 校验（Zod/JOI/class-validator）
- 防注入：ORM 参数化查询，禁拼接 SQL
- XSS 防护：CSP、输出转义、富文本白名单（如无必要禁用富文本）
- CSRF 防护：同站策略 + CSRF Token（若 Cookie 会话）
- 限流：登录/解锁/导出接口限流 + IP/设备风控
- 审计：登录失败、敏感操作、异常行为留痕
- 秘钥管理：生产环境密钥放 KMS/Vault，不入库不入代码

### 3.3 密钥与加密策略（建议）
- 记录加密算法：`XChaCha20-Poly1305` 或 `AES-256-GCM`
- KDF：`Argon2id`（从用户主密码导出主密钥）
- 密钥分层：
  - `Master Key`：由主密码派生，仅前端持有
  - `Data Key`：用于每条记录或每个用户数据集加密
- 存储策略：
  - 服务端存储密文、nonce、aad、版本号
  - 不存可直接反解明文的密钥
- 密钥轮换：支持用户修改主密码后重加密（后台任务分批）

## 4. 数据模型设计（MySQL）

### 4.1 主要表
- `users`
  - `id`、`email`/`username`
  - `password_hash`（Argon2id）
  - `mfa_secret`（加密存储）
  - `status`、`created_at`、`updated_at`
- `vault_items`（密码记录）
  - `id`、`user_id`
  - `title`（平台名，可检索）
  - `account_ciphertext`
  - `password_ciphertext`
  - `note_ciphertext`
  - `custom_fields_ciphertext`（JSON 加密后）
  - `crypto_version`、`created_at`、`updated_at`
- `tags`
  - `id`、`user_id`、`name`、`color`、`created_at`
- `vault_item_tags`
  - `item_id`、`tag_id`（联合唯一）
- `audit_logs`
  - `id`、`user_id`、`action`、`target_type`、`target_id`
  - `ip`、`ua`、`created_at`
- `refresh_tokens`
  - `id`、`user_id`、`token_hash`、`expires_at`、`revoked_at`

### 4.2 索引建议
- `vault_items(user_id, updated_at desc)`
- `tags(user_id, name)` 唯一索引
- `vault_item_tags(item_id, tag_id)` 唯一索引
- `audit_logs(user_id, created_at desc)`

### 4.3 数据分级
- 高敏感：账号、密码、备注、自定义字段（必须加密）
- 中敏感：平台名、标签名（可明文索引，必要时可做“盲索引”）
- 低敏感：时间戳、状态字段

## 5. 后端方案（Node.js）

### 5.1 技术选型
- 语言：TypeScript
- 框架：Fastify（轻量、高性能、插件生态成熟）
- ORM：Prisma（类型安全、迁移清晰）
- 校验：Zod 或 class-validator
- 测试：Vitest/Jest + Supertest

### 5.2 服务模块划分
- `auth-service`：注册/登录/MFA/会话刷新/注销
- `vault-service`：记录 CRUD、加密参数协商
- `tag-service`：标签 CRUD、关联管理
- `audit-service`：审计日志采集与查询
- `risk-service`：限流、异常检测（后续）

### 5.3 API 风格
- 请求方法统一使用 `GET` 与 `POST`
- 版本化：`/api/v1/...`
- 关键接口示例：
  - `POST /api/v1/auth/login`
  - `POST /api/v1/auth/refresh`
  - `GET /api/v1/vault-items`
  - `POST /api/v1/vault-items`
  - `POST /api/v1/vault-items/update`
  - `POST /api/v1/vault-items/delete`
  - `GET /api/v1/tags`
  - `POST /api/v1/tags/create`
  - `POST /api/v1/tags/delete`

### 5.4 安全中间件
- Helmet 安全头
- CORS 白名单
- Rate Limit
- 统一异常处理（脱敏）
- 请求 ID + 结构化日志（避免记录明文敏感字段）

## 6. 前端方案（Vue3，PC + H5）

### 6.1 技术栈
- Vue3 + TypeScript + `<script setup>`
- Vite
- Pinia（状态管理）
- Vue Router
- UI 组件框架：`Element Plus + Vant`
  - PC 端优先使用 `Element Plus`
  - H5 端优先使用 `Vant`

### 6.2 适配策略
- 单代码库响应式布局（优先）
- 断点体系：`<768`（H5）、`768-1200`（平板）、`>1200`（PC）
- 关键交互统一：列表、搜索、标签筛选、详情编辑

### 6.3 前端安全策略
- 主密码仅在运行时内存存在，不写 localStorage
- 密文数据才进入接口请求体
- Token 优先 HttpOnly Cookie（或安全存储 + 短期有效）
- 严格 CSP，禁用危险 `v-html`
- 自动锁屏：长时间无操作要求重新输入主密码

## 7. Monorepo 设计（pnpm）

### 7.1 目录结构建议
```text
repo/
  apps/
    web/                 # Vue3 前端（PC + H5）
    api/                 # Node.js 服务端
  packages/
    shared-types/        # 前后端共享类型
    crypto-sdk/          # 前端/后端共享加密协议与类型（实现分端）
    eslint-config/       # 统一 lint 配置
    tsconfig/            # 统一 TS 配置
  infra/
    docker/              # Dockerfile / compose
    sql/                 # 初始化脚本（可选）
  pnpm-workspace.yaml
  package.json
```

### 7.2 工作区与依赖治理
- 使用 `workspace:*` 依赖共享包
- 锁文件统一：`pnpm-lock.yaml`
- CI 使用 `pnpm install --frozen-lockfile`
- 通过 Changesets 管理版本（若后续拆分发布）

### 7.3 质量门禁
- Lint：ESLint + Prettier（或 Biome）
- 类型检查：`tsc --noEmit`
- 测试：单元 + 集成 + e2e
- 提交门禁：Husky + lint-staged + commitlint

## 8. 部署与运维

### 8.1 环境分层
- `dev` / `staging` / `prod` 独立
- 独立数据库与密钥
- 禁止测试数据进入生产

### 8.2 部署建议
- 前端：Nginx 静态托管（个人场景可不接 CDN）
- 后端：Docker 部署（单机或轻量云主机）
- MySQL：单实例 + 定时备份，后续按需升级高可用

### 8.3 可观测性
- 日志：结构化日志（ELK/Loki）
- 指标：Prometheus + Grafana
- 告警：登录失败激增、5xx 激增、数据库连接异常

### 8.4 备份与灾备
- MySQL 每日全量 + Binlog 增量
- 备份加密存储，异地保留
- 定期恢复演练（至少每季度）

## 11. 风险与取舍

- 采用 E2EE 会提升前端复杂度与恢复流程复杂度（如忘记主密码不可恢复明文）
- 若为了检索体验保留部分明文字段，会降低“零知识”强度
- 建议默认“高安全模式”：牺牲部分搜索体验，换取最小泄露面

---

## 12. MVP 验收标准

- 功能：记录与标签全链路可用（PC/H5）
- 安全：
  - 关键敏感字段加密存储
  - 登录安全策略生效（限流、会话、审计）
  - 通过基础渗透测试项（SQLi/XSS/CSRF/弱口令）
- 质量：核心流程测试覆盖率达到约定阈值（建议 70%+）
- 运维：具备日志、指标、告警与备份恢复能力

---
